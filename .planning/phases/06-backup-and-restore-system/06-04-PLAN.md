1. **Insert a new "## 11. Backup & restore runbook" section** at the END of `DEPLOYMENT.md` (after §10 "When something breaks"). The section should have these subsections (use `###` headings):

   ### 11.1 What gets backed up
   - Bullet list: all 26+ domain tables (via `mysqldump` of the `mysql` connection) + everything under `storage/app/public/` (profile photos + bazar receipts). One zip per backup, containing both.
   - Explicit note: `.env` is **deliberately excluded** from backups (D-07) — secrets must not live in object storage. Credential regeneration is documented in §11.5 below.

   ### 11.2 Where backups live
   - Off-server in a DigitalOcean Spaces bucket (S3-compatible). Region must match the endpoint subdomain (e.g. `nyc3` + `https://nyc3.digitaloceanspaces.com`). See §5 for the env keys.
   - Retention: daily backups kept 14 days + monthly backups kept 12 months (D-02 — the long monthly retention exists because `monthly_closings` snapshots are immutable financial records; a corruption discovered months later must still be recoverable).
   - A local copy does NOT exist — `spatie/laravel-backup` writes the zip directly to object storage. The `storage/app/backup-temp` dir is a transient working area only.

   ### 11.3 Schedule
   - **Nightly** via the Laravel scheduler (which runs every minute per §4.4). The exact cadence (set in `routes/console.php`):
     - `01:00` — `backup:clean` (prune old backups per the retention ladder)
     - `01:30` — `backup:run` (the actual DB + files dump + zip + upload)
     - `02:00` — `backup:monitor` (health-check the latest backup: not too old, not too big)
     - `03:00` — `backup:restore-test` (the periodic restore-test that proves a backup actually restores; result surfaced as a health badge in the super-admin Backups UI)
   - **On-demand** via the super-admin Backups UI (`/dashboard/backups` → "Backup now" button) or `php artisan backup:run` on the VPS.
   - **Post-close**: a successful month-close (`CloseMonthJob`) triggers an ad-hoc `backup:run` immediately. The close produces the highest-value immutable data of the month (`monthly_closings` + `monthly_member_summaries`); capturing it now beats waiting up to 24h for the nightly run.

   ### 11.4 Restore procedure (PRIMARY path — via the super-admin UI)
   Numbered steps:
   1. Log in to `/dashboard/backups` as a super-admin (the only role with access).
   2. Find the backup you want to restore in the list. Click "Restore".
   3. The restore form shows a warning + asks you to type the active mess's name EXACTLY (this is the typed-confirm second factor — D-03).
   4. Type the mess name. Submit.
   5. The app enters **maintenance mode** (web requests now 503, the queue worker is restarted so no `CloseMonthJob` runs mid-restore). The restore: downloads the zip, extracts it, runs `mysql < dump.sql`, copies the files back to `storage/app/public/`, then a spot-check verifies row counts.
   6. When complete, the app exits maintenance mode. You are redirected to the Backups UI with a success flash.
   7. An audit-log row is written (`event='backup.restore'`). Confirm under `/mess/audit`.

   ### 11.5 Restore procedure (FALLBACK — via CLI, when the UI is unreachable)
   If the app itself is down (white screen, bad migration, etc.), restore via SSH on the VPS:
   1. SSH into the VPS. `cd /var/www/mess` (or your Forge path).
   2. `php artisan down` — maintenance mode on.
   3. Download the latest backup zip from DO Spaces:
      ```bash
      php artisan tinker
      >>> $files = Storage::disk('backups')->allFiles();
      >>> $latest = collect($files)->sortDesc()->first();
      >>> Storage::disk('backups')->download($latest)->send();
      // (or use s3cmd / aws-cli / rclone outside Laravel)
      ```
   4. Unzip the backup; find the dump at `<extracted>/db-dumps/<dbname>.sql` (or glob `**/db-dumps/*.sql` if nested).
   5. Restore the DB:
      ```bash
      mysql --host=127.0.0.1 --user=root --password=<prod-password> devsroom_mess_management < <extracted>/db-dumps/devsroom_mess_management.sql
      ```
   6. Restore files:
      ```bash
      cp -R <extracted>/storage/app/public/. /var/www/mess/storage/app/public/
      php artisan storage:link   # belt-and-suspenders
      ```
   7. **Regenerate `APP_KEY` + rotate credentials** (because `.env` is NOT in the backup — D-07):
      - `php artisan key:generate` — generates a new APP_KEY. Note: existing encrypted columns (if any) will need re-encryption, but in v1 no domain columns are encrypted via APP_KEY (sessions + cookies are; users will be logged out — expected).
      - Generate fresh DB passwords + DO Spaces keys if you suspect compromise. Update `.env` + `php artisan config:cache`.
   8. `php artisan up` — maintenance mode off.
   9. Smoke-test: visit the app, log in, check `/mess/audit` shows the most recent writes.

   ### 11.6 Configure DO Spaces (one-time setup)
   Numbered steps (the operator follows these once, at first deploy):
   1. In the DigitalOcean control panel → Spaces → Create Space. Name it (e.g. `devsroom-mess-backups`), pick a region close to your VPS (e.g. `nyc3` if your VPS is in NYC).
   2. Spaces → Settings → Spaces Keys → Generate new key. Copy the access key + secret.
   3. In your VPS `.env` (via Forge's Environment UI or `nano /var/www/mess/.env`):
      ```env
      DO_SPACES_KEY=<access-key>
      DO_SPACES_SECRET=<secret>
      DO_SPACES_REGION=<region>             # MUST match the endpoint subdomain
      DO_SPACES_BUCKET=devsroom-mess-backups
      DO_SPACES_ENDPOINT=https://<region>.digitaloceanspaces.com
      BACKUP_DISK=backups
      BACKUP_MAX_MB=5000
      BACKUP_NOTIFICATION_EMAIL=ops@your-domain.com
      ```
   4. `php artisan config:cache`.
   5. Trigger the first backup: `php artisan backup:run`. Verify it landed: `php artisan backup:list`.
   6. Verify the restore-test works on the prod VPS: `php artisan backup:restore-test`. The scratch DB (`devsroom_mess_restore_test` per `DB_RESTORE_TEST_DATABASE`) is created automatically by the test.

   ### 11.7 Enable failure notifications (REQUIRED for prod)
   spatie emits `BackupHasFailed` / `UnhealthyBackupWasFound` events. The app wires these to:
   - An **in-app notification** row in the manager bell (always works; no extra config).
   - An **email** via spatie's mail channel. The default `MAIL_MAILER=log` writes the email to a log file — to actually receive the email, set:
     ```env
     MAIL_MAILER=smtp
     MAIL_HOST=<your-smtp-host>
     MAIL_PORT=587
     MAIL_USERNAME=<smtp-user>
     MAIL_PASSWORD=<smtp-password>
     MAIL_FROM_ADDRESS=ops@your-domain.com
     BACKUP_NOTIFICATION_EMAIL=ops@your-domain.com
     ```
   Recommended SMTP providers: AWS SES, Postmark, SendGrid, or your VPS's mail relay. Do NOT ship prod with `MAIL_MAILER=log` — you will not receive failure emails.

   ### 11.8 Optional: host-level snapshot (defense-in-depth)
   The spatie + DO Spaces setup is the primary backup. As a second decoupled copy (so a DO-region outage does not take both the app AND the backups), enable host-level snapshots:
   - **Forge**: site → Backups → enable (Forge writes a daily snapshot of the VPS disk).
   - **DigitalOcean**: Droplet → Snapshots → enable scheduled snapshots (daily/weekly).
   - These snapshots include `.env` (so they are sensitive — restrict access in the DO/Forge control panel). They are a full-system restore primitive, not a per-table restore — use them only when the spatie restore is insufficient (e.g. filesystem corruption outside `storage/app/public/`).
   This is OPTIONAL per the project decisions (CONTEXT.md deferred note) — Phase 6 ships "spatie + runbook", not "spatie + runbook + host snapshot". The operator can enable it independently.

   ### 11.9 Troubleshooting
   Mirror the table style of §10. Rows to include:
   | Symptom | Likely cause | Fix |
   | `backup:run` fails with `The dump process failed` (dev only) | `mysqldump` not on PATH on Windows dev | Set `DUMP_BINARY_PATH` in dev `.env` to the MySQL bin dir (e.g. `C:\Program Files\MySQL\MySQL Server 8.0\bin`). Prod Linux `/usr/bin/mysqldump` is fine. (Pitfall 2) |
   | `backup:run` fails with "authorization signature invalid" | DO Spaces region/endpoint mismatch | Confirm `DO_SPACES_REGION` matches the endpoint subdomain (e.g. `nyc3` + `https://nyc3.digitaloceanspaces.com`). (Pitfall 5) |
   | `backup:monitor` reports `UnhealthyBackupWasFound` | Backup too old OR too big | Check the scheduler is running (`sudo crontab -l \| grep schedule:run`). Check `BACKUP_MAX_MB` — increase if the mess genuinely has more data. Run `php artisan backup:run` manually to refresh. |
   | restore-test reports FAILED but the backup looks fine | Scratch DB (`devsroom_mess_restore_test`) is stale or contaminated | The test wipes it every run. If still failing, manually `DROP DATABASE devsroom_mess_restore_test; CREATE DATABASE ...;` and re-run `php artisan backup:restore-test`. |
   | restore shows "Restore failed. App is back online" but the app looks fine | The restore service caught an exception but called `up` in finally | Check `storage/logs/laravel.log` for the exception. The audit-log row (`event='backup.restore.failed'`) has the error message. The live DB was NOT modified if the exception was before the DB restore step. |
   | Restore ran but files 404 on the web (broken image links) | The `public/storage` symlink was clobbered | `php artisan storage:link`. Verify with `ls -la public/storage` → points to `../storage/app/public`. (Pitfall 4) |
   | GTID error restoring a DO-managed MySQL dump (`ERROR 3546`) | (only if you moved to DO managed DB) mysqldump emitted a `SET @@GLOBAL.GTID_PURGED` line | Self-managed VPS MySQL (per this deployment) does not hit this. If you later move to DO managed MySQL, add `--set-gtid-purged=OFF` to the dump config. (Pitfall 10) |
