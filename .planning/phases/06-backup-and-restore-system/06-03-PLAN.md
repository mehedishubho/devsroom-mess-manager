1. **`app/Http/Controllers/Backup/BackupController.php`** — mirrors `AuditController` pattern (research Code Example A). It MUST NOT contain restore logic. Structure:

   ```php
   namespace App\Http\Controllers\Backup;

   use App\Http\Controllers\Controller;
   use App\Models\RestoreTest;
   use Illuminate\Http\RedirectResponse;
   use Illuminate\Http\Request;
   use Illuminate\Support\Facades\Artisan;
   use Illuminate\Support\Facades\Storage;
   use Illuminate\View\View;
   use OwenIt\Auditing\Models\Audit;

   class BackupController extends Controller
   {
       public function index(): View
       {
           $disk = Storage::disk(config('backup.destination.disks.0'));
           $backups = collect($disk->allFiles())
               ->filter(fn ($p) => str_ends_with($p, '.zip'))
               ->map(fn ($p) => [
                   'path' => $p,
                   'size' => $disk->size($p),
                   'last_modified' => $disk->lastModified($p),
               ])
               ->sortByDesc('last_modified')
               ->values();

           $latestRestoreTest = RestoreTest::latest('id')->first();

           return view('dashboard.backups.index', compact('backups', 'latestRestoreTest'));
       }

       public function runNow(): RedirectResponse
       {
           // Ad-hoc backup. Runs synchronously here for simplicity; the UI shows
           // a "started" flash on redirect. For very large messes, this could
           // dispatch a queued job instead — but the spec is one small mess.
           try {
               Artisan::call('backup:run');
               return back()->with('status', __('Backup completed.'));
           } catch (\Throwable $e) {
               return back()->withErrors(['backup' => __('Backup failed: :msg', ['msg' => $e->getMessage()])]);
           }
       }

       public function runRestoreTest(): RedirectResponse
       {
           try {
               Artisan::call('backup:restore-test');
               return back()->with('status', __('Restore-test completed — see the health badge.'));
           } catch (\Throwable $e) {
               return back()->withErrors(['restore-test' => __('Restore-test failed: :msg', ['msg' => $e->getMessage()])]);
           }
       }

       /**
        * Super-admin-only zip download. Access-logged via a manual Audit row
        * (T-06-03: PII leak prevention — every download leaves a trail).
        */
       public function download(string $path)
       {
           $disk = Storage::disk(config('backup.destination.disks.0'));
           abort_unless($disk->exists($path), 404);

           // Audit-log the download.
           $this->writeAudit('backup.download', ['path' => $path], request: request());

           return response()->streamDownload(fn () => $disk->readStream($path), basename($path));
       }

       /**
        * Shared audit helper for the Backups surface (not a model write, so the
        * Auditable trait does not fire — manual row per research Security Domain).
        */
       private function writeAudit(string $event, array $payload, ?Request $request = null): void
       {
           $request ??= request();
           $audit = new Audit();
           $audit->fill([
               'user_type'      => $request->user() ? get_class($request->user()) : null,
               'user_id'        => $request->user()?->id,
               'event'          => $event,
               'auditable_type' => 'backup',
               'auditable_id'   => 0,
               'new_values'     => json_encode($payload) ?: null,
               'url'            => $request->fullUrl(),
               'ip_address'     => $request->ip(),
               'user_agent'     => $request->userAgent(),
               'tags'           => 'backup',
           ])->save();
       }
   }
   ```
   NOTE: read the audits table migration BEFORE finalizing — if `new_values` is a `text` column (it is per the migration), `json_encode($payload)` is the correct format (owen-it stores JSON-as-text). If the project uses a different audit schema, align.

2. **`app/Http/Controllers/Backup/RestoreController.php`** — research Code Example B. The destructive POST. Structure:

   ```php
   namespace App\Http\Controllers\Backup;

   use App\Http\Controllers\Controller;
   use App\Http\Requests\Backup\RestoreRequest;
   use App\Services\BackupRestoreService;
   use Illuminate\Http\RedirectResponse;
   use Illuminate\Support\Facades\Log;
   use Illuminate\View\View;
   use OwenIt\Auditing\Models\Audit;

   class RestoreController extends Controller
   {
       public function __construct(private readonly BackupRestoreService $service) {}

       public function show(string $path): View
       {
           $disk = \Illuminate\Support\Facades\Storage::disk(config('backup.destination.disks.0'));
           abort_unless($disk->exists($path), 404);

           $messName = \App\Models\Mess::active()?->name;

           return view('dashboard.backups.restore', [
               'path' => $path,
               'expectedMessName' => $messName,
           ]);
       }

       public function store(RestoreRequest $request): RedirectResponse
       {
           $path = $request->validated('path');

           try {
               $this->service->restoreFromDisk($path);

               // D-03 + research Security Domain: every restore writes a tamper-evident audit row.
               $this->writeAudit('backup.restore', [
                   'path' => $path,
                   'mess_name_confirmed' => true,
                   'ip' => $request->ip(),
               ], $request);

               return redirect()->route('dashboard.backups.index')
                   ->with('status', __('Restore completed. The app is back online.'));
           } catch (\Throwable $e) {
               Log::error('Backup restore failed', ['exception' => $e]);

               // Even failures get an audit row (failed restore = significant event).
               $this->writeAudit('backup.restore.failed', [
                   'path' => $path,
                   'error' => $e->getMessage(),
                   'ip' => $request->ip(),
               ], $request);

               return back()->withErrors(['restore' => __('Restore failed. App is back online; check logs.')]);
           }
       }

       private function writeAudit(string $event, array $payload, $request): void
       {
           $audit = new Audit();
           $audit->fill([
               'user_type'      => get_class($request->user()),
               'user_id'        => $request->user()->id,
               'event'          => $event,
               'auditable_type' => 'backup',
               'auditable_id'   => 0,
               'new_values'     => json_encode($payload) ?: null,
               'url'            => $request->fullUrl(),
               'ip_address'     => $request->ip(),
               'user_agent'     => $request->userAgent(),
               'tags'           => 'backup',
           ])->save();
       }
   }
   ```
   The `BackupRestoreService::restoreFromDisk()` (Plan 06-02) ALWAYS calls `Artisan::call('up')` in its finally block, so even an exception leaves the app live.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Authenticated user → /dashboard/backups/* routes | role:super-admin middleware is the first layer; the Form Request + typed-confirm + throttle are the inner layers. |
| super-admin POST → BackupRestoreService | The controller is the only caller; it writes an audit row; the service trusts its caller. |
| super-admin GET → backup zip download | Every download crosses this boundary OUT of the system (PII leak risk). Access-logged. |
| Web request → artisan restore path | The RestoreController is the only HTTP-driven path into the destructive service. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-03-01 | Elevation of privilege | `/dashboard/backups/*` routes | mitigate | `role:super-admin` route middleware on every route in the group. Verified by Test 1-6 (super-admin 200; admin/user 403; guest redirect-to-login). |
| T-06-03-02 | Tampering | POST /dashboard/backups/restore without typed confirm | mitigate | `RestoreRequest` Form Request validates `mess_name in:<active mess name>`. Verified by Tests 8-9. |
| T-06-03-03 | Tampering | CSRF on the restore POST | mitigate | `@csrf` in `_restore_form.blade.php` + Laravel's `VerifyCsrfToken` middleware (in the `web` group). |
| T-06-03-04 | Elevation of privilege | Brute-force the typed-confirm (try many mess names) | mitigate | `throttle:5,1` middleware on the restore POST route (research Security threat "brute-force typed-confirm"). The target is one fixed string so this is low-risk but defense-in-depth. |
| T-06-03-05 | Information disclosure | PII leak via backup zip download | mitigate | super-admin-only download (route middleware) + an audit-log row per download (`event='backup.download'`). Verified by Tests 12-14. |
| T-06-03-06 | Tampering | Path traversal via `{path}` route param (download/restore a zip outside the backups disk) | mitigate | `Storage::disk('backups')->exists($path)` aborts 404 if not found; the disk boundary is the guard.
