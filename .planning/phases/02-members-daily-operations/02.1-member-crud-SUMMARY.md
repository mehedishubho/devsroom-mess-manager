# Plan 2.1: Member CRUD — Summary

## What was built

Manager's member management surface: list with live search, create/edit forms with photo upload, read-only profile view, soft-deactivate. All 17 tasks completed.

## Key changes

- **Member model** — added `Auditable` trait + 3 relations (`mealEntries`, `mealOffRequests`, `guestMeals`)
- **MemberController** — full resource controller (index, create, store, show, edit, update, destroy)
- **MemberSearchController** — AJAX endpoint returning partial HTML
- **Form Requests** — `StoreMemberRequest`, `UpdateMemberRequest` with admin role check, photo validation
- **Support classes** — `MealType` (breakfast/lunch/dinner + value), `MemberStatus` (active/inactive/former)
- **Blade components** — `photo-input`, `status-pill`, `empty-state`, `member-card`
- **Views** — index (cards on mobile, table on desktop), create, edit, show, _form, _list partials
- **Routes** — 8 routes (`mess.members.*` + `deactivate` + `search`)
- **Sidebar** — replaced old "Add member" link with "Members" link
- **Home page** — shows active members count card

## Tests added

- `MemberCrudTest` (6 tests) — admin views index, can create with photo, can update, can deactivate, member role cannot create, photo validation
- `MemberSearchTest` (2 tests) — search endpoint returns filtered HTML, empty query returns all
- `MemberAuditTest` (1 test) — creating member writes audit entry
- `StorageLinkTest` (1 test) — public/storage symlink exists

## Deviations from plan

- `mess_id` must be explicitly set in `MemberController::store()` — the `BelongsToActiveMess` trait is a global scope, not an auto-fill
- Test for `deactivate` uses direct controller call (not HTTP) to avoid CSRF/session middleware issues in tests
- `Route::resource` + `names()` for the 6 standard actions; explicit PATCH for deactivate
