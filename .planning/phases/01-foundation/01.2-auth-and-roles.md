# Plan 1.2: Authentication, Roles, and Role-Based Shells

## Objective

Wire the Tyro Login package for our app: enable 2FA for the `admin` role only, route users to `/home` (manager), `/my` (member), or `/dashboard` (super-admin) after login based on their role, and ship the placeholder `/home` and `/my` Blade shells that will be enriched in later phases.

By the end of this plan: a `super-admin` user can log in to Tyro's `/dashboard`, set up 2FA, and reach the dashboard; an `admin` user is forced through 2FA setup and lands on `/home`; a `user` (member) logs in to `/my` without 2FA. Lockout still works at 5 attempts / 15 minutes (Tyro defaults).

### Task 3: Create artisan command to bootstrap a super-admin (D-22)

**File:** `app/Console/Commands/CreateSuperAdminCommand.php`

Per D-22, the user creates the first super-admin via `php artisan tinker` (or a custom command). Ship the custom command — it's clearer than a multi-line tinker call and is the documented "break-glass" path per RESEARCH.md §10.

```php
<?php

namespace App\Console\Commands;

use App\Models\User;
use HasinHayder\Tyro\Models\Role;
use Illuminate\Console\Command;
use Illuminate\Support\Facades\Hash;
use Illuminate\Validation\Rule;

class CreateSuperAdminCommand extends Command
{
    protected $signature = 'mess:create-super-admin
                            {email : Email address of the new super admin}
                            {name : Display name}
                            {--password= : Optional password (auto-generated if not provided)}';

    protected $description = 'Create the initial super-admin user (Tyro super-admin role).';

    public function handle(): int
    {
        $email = $this->argument('email');
        $name = $this->argument('name');
        $password = $this->option('password') ?? \Illuminate\Support\Str::random(24);

        $this->seedTyroRoles();

        if (User::where('email', $email)->exists()) {
            $this->error("A user with email {$email} already exists.");
            return self::FAILURE;
        }

        $user = User::create([
            'name' => $name,
            'email' => $email,
            'password' => Hash::make($password),
            'email_verified_at' => now(),
        ]);

        $role = Role::where('slug', 'super-admin')->firstOrFail();
        $user->assignRole($role);

        $this->info("Super admin created: {$user->email}");
        if (! $this->option('password')) {
            $this->warn("Temporary password: {$password}");
            $this->warn('Share securely. The user should reset it on first login.');
        }

        return self::SUCCESS;
    }

    private function seedTyroRoles(): void
    {
        Role::firstOrCreate(['slug' => 'super-admin'], ['name' => 'Super Admin']);
        Role::firstOrCreate(['slug' => 'admin'], ['name' => 'Administrator']);
        Role::firstOrCreate(['slug' => 'user'], ['name' => 'User']);
    }
}
```

Laravel 13 auto-discovers commands in `app/Console/Commands/` (no manual registration in `routes/console.php` needed).

**Verify:** `php artisan list | findstr mess` shows `mess:create-super-admin`.

### Task 4: Create artisan command to assign a role to an existing user

**File:** `app/Console/Commands/AssignRoleCommand.php`

Useful for ops during dev and for super-admin to promote themselves later. The Tyro `tyro:create-user` command creates a user but doesn't assign a role; this command is the explicit role-assignment path.

```php
<?php

namespace App\Console\Commands;

use App\Models\User;
use HasinHayder\Tyro\Models\Role;
use Illuminate\Console\Command;

class AssignRoleCommand extends Command
{
    protected $signature = 'mess:assign-role
                            {email : User email}
                            {role : Role slug (super-admin, admin, user)}';

    protected $description = 'Assign a Tyro role to an existing user.';

    public function handle(): int
    {
        $user = User::where('email', $this->argument('email'))->first();
        if (! $user) {
            $this->error("No user with email {$this->argument('email')}.");
            return self::FAILURE;
        }

        $role = Role::where('slug', $this->argument('role'))->first();
        if (! $role) {
            $this->error("No role with slug {$this->argument('role')}.");
            return self::FAILURE;
        }

        $user->assignRole($role);
        $this->info("Assigned role '{$role->slug}' to {$user->email}.");
        return self::SUCCESS;
    }
}
```

**Why the `role:admin` and `role:user` middleware:** Tyro's `EnsureTyroRole` middleware (`vendor/hasinhayder/tyro/src/Http/Middleware/EnsureTyroRole.php`) checks the user's role against the parameter. `role:admin` accepts users with the `admin` role (and the protected `super-admin` role, which is treated as a super-set — verify by reading the middleware if unsure).

**Conflict check:** `/mess/*` and `/dashboard/*` are owned by Tyro. `/home` and `/my` do not conflict with any Tyro route (confirmed in RESEARCH.md §8).

### Task 8: Role-based post-login redirect (D-02)

Tyro's `redirects.after_login` env key is a single value. We need a role-aware closure. The cleanest place to set this is a service-provider boot hook that runs **before** Tyro reads the config.

**File:** `app/Providers/AppServiceProvider.php` (replace contents)

```php
<?php

namespace App\Providers;

use Illuminate\Support\Facades\Auth;
use Illuminate\Support\Facades\Route;
use Illuminate\Support\ServiceProvider;

class AppServiceProvider extends ServiceProvider
{
    public function register(): void
    {
        //
    }

    public function boot(): void
    {
        \Carbon\Carbon::setLocale('en');

        // D-02: role-based post-login redirect.
        // - super-admin → /dashboard (Tyro admin UI)
        // - admin (manager) → /home
        // - user (member) → /my
        // - anything else → /
        config([
            'tyro-login.redirects.after_login' => function () {
                $user = Auth::user();
                if (! $user) {
                    return '/';
                }
                if ($user->hasRole('super-admin')) {
                    return '/dashboard';
                }
                if ($user->hasRole('admin')) {
                    return '/home';
                }
                if ($user->hasRole('user')) {
                    return '/my';
                }
                return '/';
            },
        ]);
    }
}
```

**Verify:** `php artisan config:show tylo-login.redirects.after_login` should show a closure.

**Note on the closure flow:** Tyro Login's `LoginController::login()` and `RegisterController::register()` call `redirect()->intended(config('tyro-login.redirects.after_login', '/'))`. `intended()` accepts a string fallback. If the closure returns a string, it's used; if it returns null, the closure is treated as a redirect hint. To be safe, always return a string.

**Edge case:** at the moment Tyro invokes the closure, `Auth::user()` is the freshly-logged-in user. Confirmed by reading `vendor/hasinhayder/tyro-login/src/Http/Controllers/LoginController.php` (the `login()` method calls `Auth::attempt()` first, then the redirect).

namespace Tests\Feature\Auth;

            'email' => 'admin@test.com',

## Notes for the implementer

- **Tyro's `RegisterController`** auto-creates users with the `user` role. This is fine for Phase 1 — public registration is enabled so anyone can self-register as a `user`, and the manager promotes them to `members` in Phase 2. D-04's manager-invite flow (replacing public registration) is in Plan 1.3.
- **`user` hasRole check vs `users` table**: `hasRole('user')` is the Tyro role-slug check, not a database-table-name collision. Verified — Tyro's `HasTyroRoles` trait provides this method.
- **`two_factor_confirmed_at`** is a column added by the `HasTwoFactorAuth` trait. Setting it in tests bypasses 2FA enforcement for the login-redirect tests (D-05's 2FA test does the inverse — verifies the gate fires when it's not set).
- **The `rate-limiter` lockout test is brittle** if other tests have already triggered lockout. The `RateLimiter::clear('tyro-login:lockout:127.0.0.1')` in `setUp()` handles it; if the key changes between Tyro versions, read the Tyro source to find the current key.
- **Tyro's `home` route** (if registered) would conflict with ours — verify by running `php artisan route:list | findstr home` after the route file is updated. If Tyro also registers a `/home`, it must be removed or our route must be loaded first. (Tyro Login does not register a `/home`; Tyro Dashboard registers `/dashboard`. No conflict.)
- **The closure for `redirects.after_login` must be a `callable`** — Laravel's `config()` helper accepts a closure as a value. If Tyro tries to `redirect()->intended($value)` and `$value` is a closure, it should be called. If Tyro's code casts to string and stores the string, the closure won't be invoked. **Verify this in code** by reading Tyro's `LoginController::redirectTo()` method. If it doesn't call the closure, fall back to a custom `Authenticated` listener that does the redirect.

- `vendor/bin/pint --test` is clean.
- `php artisan test --filter=Auth` passes (all four test files).
- A super-admin can be created via `php artisan mess:create-super-admin` and can log in to `/dashboard` after 2FA.

- A `user` (member) logs in to `/my` without 2FA.
- Lockout triggers at 5 failed attempts and redirects to `/lockout`.
- `/home` returns 403 for a `user` role; `/my` returns 403 for an `admin` role.
- The MANAGER-side `/home` view shows the manager-only "Mess settings" placeholder (Plan 1.3 will replace this with the real settings link).
