# Plan 1.1: Base Infrastructure

## Objective

Establish the foundation every other Phase 1 plan builds on: MySQL-backed schema for all 15 domain tables with `mess_id` and audit support, `Asia/Dhaka` time zone, decimal money casts, the `owen-it/laravel-auditing` package installed, and the global `mess_id` Eloquent scope wired through a small trait.

By the end of this plan: `php artisan migrate` against `devsroom_mess_management` MySQL DB creates the full Phase 1 schema; updating a `Mess` row writes a row to the `audits` table; `Mess::all()` (and every future domain model) is auto-scoped to `mess_id = 1`.

## Context

**Source docs (read first):**
- `.planning/phases/01-foundation/01-CONTEXT.md` — D-09, D-13, D-18, D-19, D-20, D-21, D-23, D-24

- `.planning/REQUIREMENTS.md` — MESS-01..04, SET-01..05, AUDIT-01..05, PERF-01, PERF-02, PERF-03, PERF-07, PERF-08, PERF-09

**Key constraints:**
- No UI in this plan. No Tyro resources, no controllers, no Blade pages. Pure infra.
- `app/` starts with just `Models/User.php` and `Providers/AppServiceProvider.php`. Everything else is created here.
- All migration filenames use Laravel's anonymous-class style, with `Blueprint` typed parameter and both `up()`/`down()`.
- All models use `#[Fillable]` / `#[Hidden]` attributes, the `casts()` method, and the new trait for mess scope.
- Decimal money: `DECIMAL(10,2)` for amounts, `DECIMAL(4,2)` for meal values. Cast with `decimal:2` in the `casts()` method.

### Task 7: Create the global mess scope infrastructure (D-20)

**Files:** `app/Models/Scopes/MessScope.php` (new), `app/Models/Concerns/BelongsToActiveMess.php` (new)

1. `app/Models/Scopes/MessScope.php`:
   ```php
   <?php

   namespace App\Models\Scopes;

   use Illuminate\Database\Eloquent\Builder;
   use Illuminate\Database\Eloquent\Model;
   use Illuminate\Database\Eloquent\Scope;

   class MessScope implements Scope
   {
       public function apply(Builder $builder, Model $model): void
       {
           $messId = config('mess.active_mess_id');

           if ($messId !== null) {
               $builder->where($model->getTable().'.mess_id', $messId);
           }
       }
   }
   ```
2. `app/Models/Concerns/BelongsToActiveMess.php`:
   ```php
   <?php

   namespace App\Models\Concerns;

   use App\Models\Scopes\MessScope;

   trait BelongsToActiveMess
   {
       public static function bootBelongsToActiveMess(): void
       {
           static::addGlobalScope(new MessScope());
       }
   }
   ```
3. **Verify by manual read-through** that the scope is null-safe (it is — `if ($messId !== null)` is the check).

**Why this approach (overriding `boot()` on every model):** single source of truth, testable, swappable for v2. The `BootBelongsToActiveMess` method is auto-discovered by Eloquent's `bootTraits()` lifecycle hook. No service-provider registration needed.

### Task 10: Factories for the 14 domain models

**Files:** `database/factories/{Name}Factory.php` for each model (14 files).

Each factory:
- Extends `Illuminate\Database\Eloquent\Factories\Factory`
- Sets `$model = App\Models\{Name}::class`
- Returns a `definition()` with realistic sample data and the required `mess_id` (use `Mess::factory()` or hardcode 1 — see notes)
- For models that have `member_id`/`user_id`/`expense_category_id` etc., use `::factory()` for the related record

**`mess_id` strategy:** because all factories run in the test's transactional wrapper, the simplest is to use `Mess::factory()->create()->id` as the default. For a faster path, you can default to `1` and let tests override — but this couples tests to migration order. Use the factory-chain approach for correctness.

## Notes for the implementer

- **No seeders run by default** (D-22: "no automated seeder for super admin / mess"). The first mess is created via the onboarding form in Plan 1.3.
- **Test data lives in factories**, not in a `DatabaseSeeder`. The `RefreshDatabase` trait wipes between tests.
- **The `Mess` model intentionally does NOT use `BelongsToActiveMess`** — it IS the mess, not scoped by one. Other models that have `mess_id` do use it. Verify in code review.
- **`Notification` model collision** with `Illuminate\Notifications\Notification` is real but manageable: the framework's trait is on `User` and uses its own class via FQCN. The domain model `App\Models\Notification` is used only by application code; Laravel's notification system still works as expected. Document in a code comment in `app/Models/Notification.php` to prevent future confusion.
- **`monthly_member_summaries.meal_rate` is `decimal:4` (not `:2`)** because meal rate precision matters at scale. Phase 3 close service requires this.
- **`Member` table allows `user_id` to be nullable** for v1 bootstrap (a member can be added before they're invited to set a password). Phase 2's invite flow will populate this.
- **`personal_access_tokens` already exists** from the Sanctum migration (D-09 from REQUIREMENTS: "Sanctum is installed but unused"). No changes to that migration.
- **The `audits` table published migration** will land with a default timestamp. Rename it to fit the project's sequence (`2026_06_16_120000_create_audits_table.php`) so it sorts correctly with the domain tables. Verify the published filename before renaming.
