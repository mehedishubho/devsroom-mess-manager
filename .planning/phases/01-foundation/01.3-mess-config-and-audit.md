# Plan 1.3: Mess Configuration, Settings, Audit Log, Member Invite, Onboarding

## Objective

Ship the user-facing slice of Phase 1: the custom app layout (top bar, sidebar, mobile drawer), mess configuration as a Tyro resource, settings as a Tyro resource, a custom `/mess/audit` Blade page, the manager→member invite flow with a signed set-password link, and the onboarding form for the first mess. All controllers use Form Requests; all views are server-rendered Blade with Tailwind v4; all user-facing strings are wrapped in `__()`.

By the end of this plan: a super-admin can create the first mess via onboarding, an admin can edit mess fields and settings via Tyro's auto-CRUD UI, an admin can view the audit log at `/mess/audit`, an admin can invite a member by email at `/mess/members/invite`, and the invited member can set a password at `/set-password?token=...` and land on `/my`.

    public function index(Request $request): View

        $query = Audit::query()
            ->with('user')
            ->latest('created_at');

        if ($model = $request->query('model')) {
            $query->where('auditable_type', $model);
        }
        if ($userId = $request->query('user_id')) {
            $query->where('user_id', $userId);
        }
        if ($from = $request->query('from')) {
            $query->whereDate('created_at', '>=', $from);
        }
        if ($to = $request->query('to')) {
            $query->whereDate('created_at', '<=', $to);
        }

        $audits = $query->paginate(20)->withQueryString();

        $models = Audit::query()
            ->distinct()
            ->pluck('auditable_type', 'auditable_type')
            ->filter()
            ->toArray();

        $users = User::query()
            ->orderBy('name')
            ->pluck('name', 'id')
            ->toArray();

        $filters = $request->only(['model', 'user_id', 'from', 'to']);

        return view('mess.audit.index', compact('audits', 'models', 'users', 'filters'));
    }

    public function show(Request $request): View|RedirectResponse
    {
        $token = $request->query('token');
        $email = $request->query('email');

        $invitation = MemberInvitation::withoutGlobalScopes()
            ->where('token', $token)
            ->where('email', $email)
            ->whereNull('accepted_at')
            ->where('expires_at', '>', now())

            'email' => ['required', 'email'],
            'password' => ['required', 'confirmed', Password::min(8)],

            ->where('email', $request->input('email'))

            ->where('expires_at', '>', now())

            'email_verified_at' => now(),
        ]);

namespace App\Http\Controllers;

use App\Http\Requests\Onboarding\CreateMessRequest;

            ['key' => 'meal_lunch', 'value' => ['amount' => (float) $data['meal_lunch']], 'type' => 'number', 'group' => 'meals', 'description' => 'Lunch meal value'],
            ['key' => 'meal_dinner', 'value' => ['amount' => (float) $data['meal_dinner']], 'type' => 'number', 'group' => 'meals', 'description' => 'Dinner meal value'],
            ['key' => 'currency', 'value' => ['code' => $data['currency']], 'type' => 'string', 'group' => 'general', 'description' => 'Currency code'],
            ['key' => 'date_format', 'value' => ['format' => $data['date_format']], 'type' => 'string', 'group' => 'general', 'description' => 'Date format'],
            ['key' => 'auto_monthly_close', 'value' => ['enabled' => false], 'type' => 'boolean', 'group' => 'general', 'description' => 'Auto-close month (reserved for v2)'],

        foreach ($settings as $row) {
            Setting::create(array_merge($row, ['mess_id' => $mess->id]));

class EnsureMessExists
{
    public function handle(Request $request, Closure $next): Response

## Notes for the implementer

- **The custom `home` and `my` views replace the Plan 1.2 inline-head versions.** Both must be deleted from Plan 1.2 (overwritten). No data loss — just a content swap.
- **`class_basename()`** is a Laravel global helper that returns the class name without namespace. Used in the audit log to show `Mess` instead of `App\Models\Mess`.
- **`@json(..., JSON_PRETTY_PRINT)`** formats the audit diff JSON for readability inside a `<pre>` block. The `<details>` element is native HTML5; no JS needed.
- **`MemberInvitation` factory** (Plan 1.1) must create a `Mess::factory()` for `mess_id`. Verify the factory chain is correct.
- **`bdt()` helper is added but not used in Phase 1 views.** It will be used heavily in Phase 3 (bill display). Keep it.
- **The audit page handles pagination via `->paginate(20)->withQueryString()`** so the filter form values persist across pages.
- **`->withoutGlobalScopes()` in `OnboardingController::create()` and `SetPasswordController::show()`** is necessary because the global `MessScope` would filter out the mess-in-question. The settings/invitations check needs to query without the scope (e.g. when `active_mess_id=1` and the mess has a different ID, or during onboarding before any mess exists).
- **The `messes` Tyro resource's `mess_id` field is NOT defined** (D-19 — `messes` is the one table that doesn't have `mess_id`). The Tyro resource config does not list it. Good.
- **The `settings` Tyro resource's `mess_id` field IS defined** as a relationship select. When creating a setting via Tyro, the user picks the mess from a dropdown. In v1 with one mess, the dropdown has one option.
- **No seeder runs by default** (D-22). The first mess is created via `/onboarding`. Tests use factories.
- **The `set-password` flow uses a plain `?token=...&email=...` query string**, not Laravel's `signed` URL middleware. The expiry check + token lookup in the controller is sufficient. Using `signed` middleware would require generating URLs via `URL::temporarySignedRoute` and would expire URLs at 5 minutes by default (too short for an email).
- **The `SetPasswordMail` template uses the `mail::message` and `mail::button` Blade components** from `laravel/framework` core (not a package). No new dependency.
