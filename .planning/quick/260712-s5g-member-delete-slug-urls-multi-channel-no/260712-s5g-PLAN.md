# Quick Task 260712-s5g: Mess management improvements batch

**Created:** 2026-07-12
**Mode:** quick (direct execution — batch is too large for the 1–3 task planner constraint)
**Branch:** master (no quick-task branch)

## Goal

Ship 7 user-requested improvements to the mess manager in one tracked batch:
member delete, name-based member URLs, multi-channel notifications, duplicate
prevention, role-based sidebar, a whole-app audit, and a README refresh.

## Scope (7 items)

### 1. Delete mess member
- `MemberController::destroy` currently only deactivates (sets status=inactive).
- Add a real delete: `DELETE /mess/members/{member}` → soft-delete (`$member->delete()`).
- Add super-admin-only permanent delete: `DELETE /mess/members/{member}/force` →
  `forceDelete()` guarded by a dependency check (blocks if payments/meals/expenses
  exist — protects referential integrity).
- Keep the existing `PATCH /mess/members/{member}/deactivate` (status-only, used by
  the meal grid denominator logic).
- UI: add Delete + Permanently-delete buttons to the member show page with confirm.
- **Files:** `MemberController.php`, `web.php`, `show.blade.php`.

### 2. Name-based member URLs (unique slug for duplicates)
- Migration: add nullable `slug` (varchar 255) to `members`; backfill existing rows.
- `Member::getRouteKeyName()` → `'slug'` (Laravel-idiomatic; all `{member}` routes
  resolve by slug and `route(..., $member)` emits `/mess/members/{slug}`).
- Unique-slug generator: `Str::slug($name)`; collisions in the same mess get `-2`,
  `-3`, … then a 4-char random suffix if still taken. Regenerate on name change.
- **Files:** new migration, `Member.php`, `MemberController.php` (store/update).

### 3. Multi-channel notification system (email / WhatsApp / Telegram / phone/SMS)
- Store per-mess config in the existing `settings` table under group `notifications`.
- `App\Contracts\NotificationChannel` interface + 5 implementations: InApp (existing),
  Email (Laravel Mail), Telegram (Bot API HTTP), WhatsApp (Twilio/Meta HTTP),
  Sms/Phone (Twilio/Vonage HTTP). Each fails open (logged, never crashes the caller).
- `ChannelManager`: resolves enabled channels for the active mess + notification type,
  resolves recipient contact, dispatches each.
- Extend `NotificationService::send()` / `broadcastToManagers()` to fan out to enabled
  channels in addition to the in-app record.
- Admin UI: new `/mess/notifications` page (role: admin/super-admin) — per-channel
  toggle + credentials, per-type routing matrix. Added to sidebar "Settings" group.
- **Files:** interface, 5 channels, `ChannelManager`, `NotificationService`,
  `MessNotificationController` + request, settings view, `web.php`, sidebar.

### 4. Duplicate member prevention
- `StoreMemberRequest` + `UpdateMemberRequest`: `email` and `mobile` unique per mess
  (where present), scoped to non-soft-deleted rows.
- **Files:** `StoreMemberRequest.php`, `UpdateMemberRequest.php`.

### 5 + 7. Role-based sidebar reorganization
- Current sidebar is a flat list in `layouts/app.blade.php` shown to everyone — members
  see a full row of 403'd manager links.
- Extract to a structured partial `components/sidebar.blade.php` with grouped sections
  (Mess / Finance / Reports / Closing / Settings / System) and role gating:
  - super-admin → everything + System (Admin dashboard, Backups, Notifications config, Audit).
  - admin/manager → Mess + Finance + Reports + Closing + Settings (Notifications, Mess settings).
  - user/mess-member → My-section only (My dashboard, My bill, My payments, My reports).
- Section headers + active-state logic preserved; min-h-[44px] touch targets preserved.
- **Files:** `components/sidebar.blade.php`, `layouts/app.blade.php`.

### 6. Whole-app audit — recommendations
- Written audit (add/remove/rearrange) captured in `260712-s5g-RECOMMENDATIONS.md`
  and summarized in SUMMARY.md. Drives any small include/remove wins feasible in-scope.

### 7. README refresh for public release
- Update Key Features (delete, slug URLs, multi-channel notifications, duplicate
  prevention, role-based sidebar), add a "Role-based access" overview table, refresh
  Tech Stack if needed, keep the existing professional structure.
